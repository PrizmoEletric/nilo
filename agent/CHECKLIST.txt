KNOWLEDGE BASE — SQLite (NEXT MAJOR FEATURE)
────────────────────────────────────────────────────────
Objective: replace all JSON knowledge files with a single SQLite database
so Nilo can learn, query, and reason about his world autonomously.

WHY SQLite OVER JSON:
  - Multiple JSON files (modded-state-ids.json, modded-blocks-mapping.json,
    learned-physics.json) are siloed and can't reference each other.
  - As Nilo learns more (blocks, entities, locations, player corrections),
    querying and joining across topics requires a real database.
  - better-sqlite3 is synchronous and zero-config — perfect for a game bot.
  - One file on disk instead of many scattered JSON files.

SCHEMA (planned):
  blocks     — name, bounding_box, is_solid, is_passable, hardness,
               source (vanilla/auto/player/observed), confidence, taught_by
  state_ids  — state_id → block_name, source, confidence
               (replaces modded-state-ids.json + modded-blocks-mapping.json)
  entities   — name, is_hostile, source, confidence, taught_by
  locations  — x, y, z, label, notes
  learned    — full history of what the player has taught Nilo and when

PHYSICS RESOLUTION (replaces hardcoded BLOCK_PHYSICS table):
  When creating a descriptor for a state ID, check blocks table first.
  If found → use stored physics. If not → look up vanilla registry by name.
  If still not found → heuristic fallback + mark as low confidence.
  Truly modded blocks with unknown physics can be taught conversationally:
    "yigd:grave is solid" → saved to blocks table, applied immediately.

CONVERSATIONAL LEARNING (extends current block alias command):
  "pumpkin_stem is stone_bricks"  → remap state IDs + inherit vanilla physics
  "yigd:grave is solid"           → save to blocks table as player-taught
  "that creeper mod is hostile"   → save to entities table
  All saved to DB with source='player', taught_by=username, timestamp.
